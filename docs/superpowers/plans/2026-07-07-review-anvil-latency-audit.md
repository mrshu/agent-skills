# Making review-anvil faster without compromising quality

Final recommendation report. All 16 proposals were run through independent quality-skeptic and feasibility-skeptic passes; every proposal survived, but 14 of 16 required amendments, and several had major components cut. The recommendations below are the **amended** versions — where the two skeptics diverged, the stricter constraint wins.

---

## 1. Diagnosis: where the wall-clock actually goes

From the latency maps and ~30 mined real sessions (Jun–Jul 2026), a typical 3-round `per_fix` run spends its time in five places, in order:

1. **Reviewer waves** — 3+ rounds × max(reviewer latency), ~100–215s typical per wave, pinned at 630s (timeout+kill grace) whenever one reviewer hangs. Rounds are strictly sequential (barrier B6) and each wave waits on its slowest member (B2). This is the largest fixed serial cost.
2. **`verify_cmd` suite runs** — baseline + per-round gate + per-deletion runs, uncapped, fully serial today. On slow suites this dominates everything.
3. **The batched reproduction verifier** — one reviewer-scale call per round with candidates, strictly serialized between wave and fix application (B3).
4. **Tail work** — the adversarial gate (read-only runs; observed hanging *forever* in production), then 1–5 minutes of LLM report/inline-body composition, all post-loop and serial.
5. **Pure waste, not design** — the biggest observed losses are failure modes, not architecture: stale local base refs producing bogus +2377-line diffs and whole re-run rounds; the host's 2-minute default Bash timeout SIGKILLing healthy reviewer waits (exit 143); codex "at capacity"/stdin-hang failures with a hard no-retry policy; mid-run author pushes making the final round's entire fix/gate/report chain redundant; dead ScheduleWakeup turns after completion; no resume after interruption.

Two structural facts shape everything below: the skill is **prose-orchestrated** (every change must be executable by an LLM following SKILL.md, so mechanisms must be mechanical, not judgment-dependent), and the **reproduction pass is the proven quality backbone** (6 false positives refuted in one mined run) — nothing here cuts it; two items make it faster without weakening it.

The single most important cross-cutting finding from verification: several proposals originally assumed test suites and reviewers could share a worktree concurrently. They cannot — reviewers legitimately execute builds/tests, suites mutate trees, and a collision can flip a gate spuriously red and revert *good* fix commits. Every overlap recommendation below therefore preserves a **one-suite-at-a-time / isolated-checkout invariant**.

## 2. Recommendations, ranked by expected latency impact vs effort

### Tier 1 — quick wins (small effort, high or highly certain savings)

**R1. Mandate Bash tool timeout ≥ reviewer_timeout + 90s on any call that waits on reviewers (M8, amended). Effort: small (doc-only).**
*Change:* Add one rule to engine `SKILL.md` §2 (lines 117–168): any Bash call that can block on a reviewer — the background-and-`wait` fallback (line 143), the serial last resort (line 166), inline wrapper replication (line 159) — must set the host Bash tool's own `timeout` to at least `reviewer_timeout + 90` (wrapper deadline + 30s TERM→KILL + margin). On hosts with `run_in_background`, forbid blocking foreground waits outright. If the host max is too low, prefer detached dispatch + short polling calls over reducing `reviewer_timeout`; reduction is last-resort, floored at 300s, and **forbidden** on >5000-line diffs (where the timeout is deliberately doubled). Add a Failure-handling row: exit 143 / missing `STATUS=` line in wrapper stdout = "host tool timeout — misconfigured dispatch, not a reviewer failure"; re-dispatch exactly once after fixing the dispatch mode. Do **not** key this on the `.timedout` stamp — it never persists and is never created in this scenario.
*Why safe:* Pure infrastructure correctness; today these kills are misclassified as reviewer failures, silently losing a healthy reviewer's pillar coverage (no retries) or aborting adaptive rounds. This is a quality *gain*.
*Saving:* ~3–5 min per bitten round; removes the failure class entirely on non-Agent hosts, where it bites most waves (98–213s reviewers vs a 120s default).

**R2. Parallel adversary dispatch + hard deadlines + one-shot-alarm/no-polling rule (M4, amended). Effort: small.**
*Change:* (a) Adversarial section (`SKILL.md` ~lines 235–254): launch all adversaries of the selected mode in a single multi-tool-call background message, same rule as §2 reviewers; synthesize verdicts when all return or hit deadline. (b) Every Bash-dispatched reproduction verifier and adversary runs under `run-reviewer.sh` with cap = the **effective** `reviewer_timeout` (after the >5000-line doubling — do not clamp to 600s, which would starve adversaries on exactly the runs that mandate full mode). Agent-tool dispatches, which have no timeout and are the source of the observed infinite "Awaiting its verdicts" hang, get exactly one one-shot deadline alarm per dispatch wave (`sleep <cap>` as a background Bash task, killed when agents return). On deadline: salvage complete per-ID verdicts already in the output file, then apply the existing Edge Cases rows to the rest (verifier failure → safe-side Deferred; adversary failure → continue and note; strict → COMMENT). Record "timed out at \<cap\>s" in Run Details. (c) Concurrency rule: forbid recurring/polling wakeups; permit only the one-shot alarm; after printing the final report path (engine §6 final-report step 4), cancel all remaining alarms/monitors.
*Why safe:* Adversaries are independent by design (each attacks the same frozen ID set); degradation paths reuse already-written semantics and are strictly conservative (more cautious output, never less verified); verdict salvage prevents one slow candidate from deferring finished confirmations.
*Saving:* 2–7 min off the tail of every read-only PR run with adversarial enabled (the `review-anvil-pr` default), and converts unbounded hangs into a bounded ≤cap wait. Also kills the observed dead post-completion wakeup turns.

**R3. Base-freshness fetch + mid-run head-drift guard (M6, amended). Effort: medium, but each piece is small.**
*Change:* (a) For branch/base targets, **always** `git fetch <remote> <base>` before materializing the diff and diff against the remote-tracking ref (fetch alone does not move a local `<base>` branch). For improve-pr, implement in `pr-helper.sh verify-checkout`: make the conditional fetch (lines 1064–1075) unconditional and always emit the remote-tracking form as BASE_BRANCH. Do not use commit-age as a freshness proxy — it measures nothing about staleness vs the remote. (b) Round-1-only sanity cross-check against `gh pr view --json files`, direction-aware (stale base = local diff contains many files *absent* from GitHub's list; local-only extras from unpushed fix commits are expected), skipped when GitHub reports ≥100 files (API truncation); on mismatch, refetch and re-snapshot once, then proceed and note the discrepancy rather than blocking. (c) Per-round drift probe in improve-pr: one `gh pr view --json headRefOid` at each snapshot; if the remote head is neither the recorded HEAD_SHA nor an ancestor of local HEAD, the author pushed mid-run — stop **before** dispatching the wave and surface it via the existing failure path (local commits preserved). Never auto-rebase the run's fix commits or auto-resolve conflicts.
*Why safe:* Reviewers see the real diff instead of a stale-base artifact — this is quality-positive (mined sessions show stale bases produced *false high-severity findings*). The drift stop converts a doomed push into an explicit early pause.
*Saving:* Kills the two most expensive observed waste modes: 5–20 min whole-round re-runs from bogus diffs, and entire redundant final rounds ending in a rejected push. ~2–5s/round cost.

**R4. Overlap the verify_cmd baseline with round-1 reviewer dispatch (M1, amended). Effort: small.**
*Change:* `per_fix` only. Resolve `verify_cmd` (explicit / none / auto-detect, per `fix-application.md`) as an explicit **pre-dispatch** step — today resolution lives in a file read mid-round — then launch the baseline run as a background Bash job in the same single multi-tool-call message that dispatches round-1 reviewers. Collect its result **before Loop Mechanics §3 dispatches the reproduction verifier or executes any deletion-execution test run** — not §4, because §3's deletion red/green judgment consumes the baseline, and joining at §4 could put two suite instances in flight at once on slow suites. Add one line to round-1 reviewer context: "A baseline test run is in progress; do not execute the project test suite or mutating build commands this round." When the job completes, run `git status --porcelain`: if the suite mutated tracked files (codegen-style suites), restore the tree, fall back to serial baselines for this repo, and mark round-1 findings anchored in mutated files as mandatory reproduction candidates. `post-start` stays serial and pre-engine as today (see Rejected).
*Why safe:* The baseline still gates round-1 fixes at the same decision point; reviewers never consume it; the one-suite-at-a-time invariant and a settled baseline for the §3 deletion judgment are preserved; the mutation hazard is now detectable rather than assumed away.
*Saving:* One full test-suite runtime off the pre-round-1 critical path, hidden under the ~100–215s (up to 630s) wave — on slow suites, the entire head-stage cost, once per run.

**R5. Small-diff reviewer_timeout of 420s, requested rounds only (M13, amended). Effort: small.**
*Change:* `SKILL.md` Parameters table (line 41) + §2: keep 600s default and the >5000-line doubling, but when the raw diff is under ~500 lines, the unset default becomes **420s** (~2× the slowest legitimate reviewer observed, 213s — not the originally proposed 300s, which is only 1.4× a heavy-tailed max). Two hard exclusions: user-set timeouts are never scaled, and **adaptive rounds always use the full 600s** — a false timeout there aborts the whole round before fixes (lines 401–404), an asymmetric cost. No script change; the timeout is already a per-invocation argument.
*Why safe:* Timed-out reviewers already degrade-and-proceed in requested rounds; 2× headroom over the observed max; the round outcome for a genuinely hung reviewer is identical, just 3 minutes sooner.
*Saving:* Pinned small-diff rounds drop from ~630s to ~450s worst case; ~9 min across a 3-round run with one hang per round; zero cost on healthy runs.

**R6. Scale the adaptive-continuation budget and require new findings (M12, amended). Effort: small.**
*Change:* (a) Per_fix default becomes `max_rounds = min(max(6, rounds), rounds + adaptive_budget)` with budget 1/2/3 for small/medium/large diffs (measured at target resolution — per_fix targets are always local git diffs). The `min()` with the old formula guarantees a **pure reduction**: runs with `rounds ≥ 6` gain nothing new. "Continue if needed" restores the legacy `max(6, rounds)` cap as a one-phrase opt-out. Update **every** hardcoded "6": engine `SKILL.md` lines 28, 52, 53, 80 and `review-anvil-improve-pr/SKILL.md` lines 114, 126 — stale numbers will misdirect a prose orchestrator. (b) §6 continuation gains the criterion "latest round produced ≥1 NEW material finding", where new **includes** a reproduction-confirmed re-detection of a previously fixed-and-verified finding (evidence the fix failed — the core case adaptive continuation exists for) and excludes only unconfirmed/refuted re-detections, already-deferred items, and repeated-never-fixed findings (already excluded today). (c) Extend the tuning-suggestion rule to name the tier and the exact larger cap to re-run with.
*Why safe:* The line-412 tuning-suggestion backstop means the failure mode is "user is explicitly told to continue," never a silent miss; the new-findings criterion mostly codifies existing intent with the incomplete-fix case explicitly protected.
*Saving:* Tail cap, not mean: removes up to 2 potential extra round chains (3–10+ min each) on small/medium diffs.

**R7. Dispatch codex-exec with stdin from /dev/null (M16 Part A). Effort: trivial.**
*Change:* `SKILL.md` §2: codex-exec's prompt arrives as argv; the wrapper's `<&0` stdin passthrough exists only for the claude fallback and is what exposes codex to the observed stdin-hang class. Redirect codex dispatches from `/dev/null`.
*Why safe / saving:* Root-cause elimination of the most common hang class at zero cost. (Part B, stall detection, is Tier 3 — it depends on R8.)

### Tier 2 — medium effort, solid returns

**R8. One bounded retry for fast-failing reviewers (M7, amended). Effort: medium.**
*Change:* Amend `SKILL.md` Failure handling (line 453 **only** — the "unparseable findings" edge-case row is a successful full-length run the wrapper can't see and must stay no-retry): a reviewer returning STATUS=failed/empty with elapsed under min(30s, 25% of reviewer_timeout) — the signature of codex capacity errors and stdin loss vs a legitimate 98–213s review — is re-dispatched exactly once. State the rule generically so it also covers Agent-tool reviewers. Implement in-wrapper for Bash dispatches: `RETRIES=1` in `run-reviewer.sh`, buffering the piped prompt to a temp file beside the out-file so stdin is re-suppliable; **attempt 2 gets only the remaining budget** (total wall clock across both attempts ≤ original timeout; a mid-retry deadline classifies as timeout, never retried again); preserve attempt 1's stderr as `<out>.err.1`; print `RETRIED=1` and require the round summary to note it, so backend flakiness stays visible. Same rule applies in adaptive rounds (retry-then-ok = normal completion; retry-exhausted = existing abort). Update `test-run-reviewer.sh` with cases for byte-identical prompt delivery on both attempts (the `<&0` rebind is load-bearing), failed-then-ok, empty-then-ok, failed-twice, timeout-never-retried.
*Why safe:* Strictly recovers coverage that is silently lost today (lenses partition — a dead reviewer loses a pillar outright); every existing conservative backstop still fires on double failure. STATUS=timeout is never retried, so a slow reviewer can't double the wave.
*Saving:* Eliminates the two costliest recovery paths: manual end-to-end wave re-runs (+2–11 min) and adaptive-round aborts that discard a fully executed wave. The retry runs while peers are still working, so it's near-free on the critical path.

**R9. Small-diff agent count: agents=2 (M11, amended — rounds tiering dropped). Effort: low-medium.**
*Change:* Add a "Run sizing" step at the top of Loop Mechanics §1, after the round-1 diff is materialized (not at Parsing — mix resolution and backend validation depend on it). Classify using the **existing** adversarial-auto meaningful-changed-size policy (excludes generated/vendor/lockfile churn): SMALL = <~200 meaningful lines, ≤5 files, and no risky surface — judged over the diff *hunks* (auth, migrations/schema, concurrency, public API/serialization, money/time), not filenames; any uncertainty → MEDIUM. SMALL sets only the **unset** `agents` default to 2 (1 codex + 1 claude; the M=2 lens pack covers all four pillars). Do **not** tier rounds/max_rounds (see Rejected). M=2 failure escalation: if a reviewer fails, re-dispatch one replacement once; a round that ends with one reviewer's output is marked degraded coverage, **cannot satisfy convergence early-exit**, and the next round runs agents=3. Explicit user values always win. Log tier and effective defaults in Run Details.
*Why safe:* Each reviewer reads the whole <200-line diff; because consensus is structurally rarer at M=2, *more* single-reviewer findings route through the reproduction gate — the false-positive control tightens. The degraded-round rule stops a lone clean flag from ending the loop.
*Saving:* Slowest-member exposure drops from max-of-3 to max-of-2 every round (~0.5–2 min/round typical; up to ~7 min when the dropped slot would have hung), plus a third less synthesis/dedup input.

**R10. Chunk the reproduction batch by materiality, dispatched in parallel (M15, amended — cap-and-defer replaced). Effort: medium.**
*Change:* `SKILL.md` §3 (lines 178–204): order candidates into Tier 1 (deletion/dead-code/high-blast-radius, all critical/high including inferred-evidence, orchestrator-uncertain high/critical — **never** capped or deferred, since deletions and required high-risk candidates cannot become actionable without reproduction) and Tier 2 (single-reviewer and uncertain mediums). Split into ~10-candidate chunks and dispatch all chunk verifiers **in parallel in one multi-tool-call message** (same pattern as reviewer dispatch, already sanctioned by line 190's split rule and the within-round parallelism rule) rather than one monolithic serial batch. Every candidate still gets a verdict; a per-chunk failure degrades only that chunk via the existing safe-side path. Pathological backstop only: ceiling of ~4 parallel chunks; beyond ~40 candidates, Tier-2 overflow takes Deferred "batch cap — unconfirmed" with the cap noted in the round summary. Update `reproduction-prompt.md`'s dispatch note; re-run `test-reproduction-policy.sh` per the existing policy-change rule.
*Why safe:* Identical wall-clock to a hard cap (one chunk's runtime) but no candidate loses its confirmed/refuted verdict in the normal case — the FP-refutation function (6 FPs refuted in one mined run) is fully preserved; the extra cost is tokens, not latency.
*Saving:* Bounds the second-largest serial cost: finding-heavy round-1 verifier calls stop ballooning; minutes on heavy rounds.

**R11. Close the refuted-verdict leak + delta focus for later rounds (M10, amended — headline saving corrected). Effort: low-medium.**
*Change:* (a) The PRIOR ROUNDS block already suppresses addressed/deferred re-raises, but **refuted** findings are dropped from final Findings and never enter it — so round-N≥2 reviewers re-raise them and force re-verification. Fix in two layers: add a `refuted:` list (with one-line evidence) to PRIOR ROUNDS plus the do-not-re-raise paragraph; and at candidate-building time, semantically match candidates against prior refuted/unclear verdicts (same standard as the dismissed check, line 182) and reuse the verdict — invalidated via `git diff --name-only <verdict_sha>..HEAD` intersected with **anchors ∪ evidence-cited files** (a refutation resting on "caller X guards this" must be invalidated when the evidence file changes, not just the anchor). Never cache `unclear`; never cache verdicts citing no concrete file; a re-raise carrying new evidence or arguing the prior verdict is wrong bypasses the cache (preserves the prompt's existing escape hatch). In `commit_mode=none` multi-round runs (identical baseline) cache unconditionally. (b) Per_fix rounds N≥2 get a CHANGED SINCE LAST ROUND block (diff of round N−1's fix commits): prioritize changed hunks, callers, previously flagged areas, then sweep the rest; each reviewer must end with `sweep: completed | partial`, and a round with any `partial` **cannot** classify as clean/nits_only for convergence early-exit.
*Why safe:* Evidence-aware invalidation forces re-verification exactly when fix commits touched verdict-relevant files; sweep honesty prevents clean-by-omission from corrupting the stopping condition.
*Saving:* Honest estimate ~0–3.5 min/run (occasionally skipped verifier dispatch whose only candidates were refuted re-raises) plus less synthesis churn; delta focus is primarily a quality/focus improvement with possible latency upside.

**R12. Speculatively dispatch round N+1 during round N's gate — isolated gate only (M3, amended). Effort: medium.**
*Change:* Requested rounds only, while `completed_rounds < rounds`, and only when round N's convergence flag is `material_findings` (early exit and adaptive rounds are decidable at synthesis and are excluded — adaptive continuation *requires* the gate result). Hard precondition: the gate runs in a **disposable isolated checkout** (`git worktree add <scratch> <post-fix HEAD>`), never the worktree reviewers read — proven once cheaply by running the R4 baseline in the same isolated mode; if isolation fails (suites needing in-tree deps/env), fall back to today's sequential behavior. Sequence: synchronously materialize round N+1's snapshot, then one multi-tool-call message launching the backgrounded gate and the reviewer wave, with every speculative output file stamped `round<N+1>-attempt<k>-<label>@<sha>.md`. The N+1 prompt's PRIOR ROUNDS header reads `verification pending — gate running` (the template demands a verification state that isn't known yet; fabricating "passed" is the alternative). Gate green: proceed normally. Gate newly red: kill the wave (best-effort), delete attempt-k files, apply the existing fix-forward/revert rule; if revert fails to restore green, **stop the loop** per existing failure handling — do not re-dispatch. Discard is enforced mechanically: before N+1 synthesis, verify HEAD equals the stamped SHA — a stale wave is invalid even if the kill was missed (late completion notifications carrying an old attempt suffix are ignored). Amend the Concurrency section to name this as the single sanctioned inter-round overlap.
*Why safe:* The original same-worktree design could flip the gate spuriously red (reverting *good* fixes) or feed reviewers a mid-mutation tree; isolation removes both mechanisms. The SHA/attempt stamp makes "synthesize from stale reviews" impossible by construction — important, since transcript mining shows orchestrators fumble exactly this kind of conditional bookkeeping. Misprediction cost is one wasted wave (tokens).
*Saving:* Per-round, min(suite runtime, wave duration) for rounds 1..rounds−1 — with R4, this takes `verify_cmd` off the critical path everywhere except the final gate. Honestly narrower where suites can't run isolated (those keep today's behavior at zero quality change).

### Tier 3 — larger or riskier; do after the above proves out

**R13. Overlap reproduction verification with consensus fix *editing*, committing after verdicts (M2, amended). Effort: medium.**
`per_fix` only (in `commit_mode=none` §4 is skipped — nothing to overlap). Pin a snapshot commit at §1 (`HEAD` when clean; `git stash create`/temp-index `commit-tree` when the tree is dirty — a bare `rev-parse HEAD` omits uncommitted targets), materialize a detached read-only worktree, and point the batched verifier at it with the primary checkout's dependency state bridged (invoke the primary venv/node_modules binaries with the worktree as cwd); if the snapshot can't be constructed reliably, keep the sequential barrier — never point the verifier at the mutating live tree. While it runs, the orchestrator may **edit (not commit)** fixes for 2+-consensus non-candidates whose *planned edit set* (not just cited files, including `file_alternates` anchors) is disjoint from every candidate's files, in a different fix group, sharing no root cause. All commits happen after verdicts land, preserving the verdict-before-commit invariant — a refutation that implicates a prepared fix can still amend or drop it. The gate runs once over the whole round's fixes and reverts all of them on red. Remove the worktree in the same step that applies verdicts. Saving: hides the 1–3 min/round verifier behind fix work; zero in rounds where everything is a candidate.

**R14. Pre-draft report/inline bodies during background waits, with mechanical invalidation (M5, amended). Effort: medium.**
The two skeptics split on scope: the quality pass restricted drafting to per_fix terminal findings; the feasibility pass showed the heavy body tail actually lives in single-round `review-anvil-pr` (the only flow posting `.inline.json`), whose dead-time windows are the reproduction and adversarial waits. Reconciliation: drafting is allowed in both, but **all drafts are provisional** and the §6 reconcile never trusts them — each draft records ID, severity, verdict, anchor, every cited file, and a wording hash; a draft is used verbatim only if final severity/verdict/wording match, no post-draft commit or verdict touched any cited file, and the anchor resolves against the final diff; anchors and suggestion fields are always re-validated regardless. Anything touched by reproduction (refuted/downgraded/narrowed) or adversarial verdicts (drop/defer/modify, suggestion stripping) is redrafted from final state. Drafts go to a working file, never restated in the transcript; drafting yields immediately on any completion; skip entirely under ~4 findings (the reconcile machinery would cost more than the tail). In the non-Agent host fallback, sequence drafting between launching wrappers and the blocking `wait`. Note there is no "Phase F" — the tail is Loop Mechanics §6 + Output Format. Saving: cuts the 1–5 min LLM tail to a reconcile-and-write pass on finding-heavy runs; also surfaces oversize bodies early. Ranked Tier 3 because it adds a remember-to-reconcile step exactly where late-run instruction drift bites.

**R15. Confirmation-round tier: reduced wave with one full-scope reviewer (M14, amended). Effort: medium.**
`per_fix` only. When the next round follows one where all material findings were fixed, verification state was strictly `passed` (not skipped/none-detected/pre-existing), and remaining items are below `min_fix_severity`: dispatch min(agents, 2) — exactly **one** reviewer fix-hunk-scoped, the **other** full-scope with normal lens priorities (full diff to both; the M=2 lens row from `reviewer-prompt.md`; cross-backend when the run had both). This preserves whole-target breadth in the round whose clean signal can terminate the run — the reproduction pass refutes false positives, never false negatives, so nothing else audits an absence of findings. Mechanical full-wave disqualifiers, inlined (no dependency on R9): fixes touched shared state/concurrency/public API/serialization or any risky surface; the round is an adaptive round justified by the "risky/shared surface" criterion (it exists to catch second-order bugs *outside* the fix hunks); any fix commit >~40 lines or fixes spanning >3 files; the user pinned an explicit mix (SKILL.md: "honor a mix exactly"). Any material finding reverts the next round to the full mix. Record "confirmation round (reduced wave: 1 full-scope + 1 fix-scoped)" in the round summary *and* Run Details, so early exits decided by a reduced wave are auditable. Saving: 1–4 min per tail round. The riskiest item in this set — ship last among the loop changes.

**R16. Checkpoint + resume, restricted to tamper-evident artifact reuse (M9, amended). Effort: large.**
State lives **outside the repo under review** (host scratchpad or `$TMPDIR/review-anvil/<repo-hash>/<run-id>`) — the repo is attacker-writable and a repo-resident `state.json` re-read by a fresh session is a persistence channel that bypasses the consensus/reproduction gates (fabricated 2-reviewer out-files skip reproduction entirely); baseline-SHA matching gives zero protection for untracked files. Record a SHA-256 per reviewer out-file. Checkpoint at three points: post-snapshot (baseline identity: **PR head SHA** for PR targets, since local HEAD is informational-only; HEAD otherwise), at wave-return before synthesis (persisting Agent-tool reviewers' in-band output to disk — otherwise resume recovers only the codex fraction; never infer ok from a non-empty file, since killed sessions leave truncated ones), and at round-summary. Mark the state file **terminal when the final report is written** — otherwise a completed read-only run (HEAD never moves) would hijack the next legitimate re-run. Resume gate: exact target-state match per target type (diff hash for uncommitted targets — HEAD alone is never sufficient; clean worktree + fix-commit lineage for mid-run per_fix), <24h, all hashes verify, state and out-files untracked and absent from the baseline commit. Resume reuses exactly two things: STATUS=ok reviewer outputs, and wholly completed per_fix rounds whose fix commits are ancestors of HEAD with a green gate (git history is the durable, tamper-evident record). **Judgments are never resumed**: synthesis, the dismissed check (with a freshly re-fetched list), reproduction, adversarial, and the gate always re-run. Ship with a fixture test (dirty-tree refusal, tampered-hash refusal, moved-PR-head refusal, terminal-state refusal, straggler-only re-dispatch). Recovers ~60–85% of an interrupted round's cost; zero effect on uninterrupted runs.

**R17. Stall detection for codex dispatches (M16 Part B). Effort: medium. Depends on R8.**
Only after R7's `/dev/null` fix and R8's retry land: extend the `run-reviewer.sh` watchdog into a 60s poll loop with opt-in `STALL_SECS` (default off) sampling combined `wc -c` of out+err; 0 bytes after ~180s → TERM, `STATUS=stalled`/exit 125, retried once via R8. Codex-exec only (claude `-p` legitimately writes nothing until the final answer). Two binding preconditions: SKILL.md recommends enabling it only where R8's retry is active (otherwise a false kill converts a possibly-healthy reviewer into permanently lost coverage), and first-byte latency must be empirically confirmed **under the wrapper's file-redirected non-TTY stdio** — progress streaming observed on a TTY may be suppressed or block-buffered when redirected, and if codex goes quiet under redirection every healthy >180s run would be falsely killed. Adaptive rounds: retry-ok = normal; retry-failed = existing abort (stall detection there makes a slow abort fast, nothing more). Update the status contract, failure-handling list, and `test-run-reviewer.sh`. Saving: residual hangs drop from a 600–1200s pin to ~180s + concurrent retry.

## 3. Suggested implementation order

**Wave 1 — doc-only / trivial, land immediately, independent of each other:**
1. R1 (Bash timeout rule) — pure prose, removes an active failure class.
2. R7 (codex `/dev/null` stdin) — one-line dispatch change.
3. R2 (parallel adversaries + deadlines) — small prose + wrapper usage; biggest tail win for the default PR preset.
4. R5 (420s small-diff timeout) and R6 (adaptive budget) — small prose edits; R6 must touch all six hardcoded "6"s in one commit.

**Wave 2 — head/waste elimination:**
5. R3 (base freshness + drift guard) — `pr-helper.sh` + prose; highest per-incident payoff in the set.
6. R4 (baseline overlap) — small; also proves the isolated-gate pattern R12 needs.
7. R8 (fast-fail retry) — wrapper + tests; **prerequisite for R17** and strengthens R9's M=2 escalation.

**Wave 3 — loop restructuring (each builds on Wave 2's guarantees):**
8. R9 (agents=2 sizing) — its tier classification is also reusable by R5/R6 if you want to unify sizing.
9. R10 (chunked reproduction) and R11 (refuted-leak fix) — independent; both shrink the §3 serial segment.
10. R12 (speculative dispatch) — requires R4's isolated-gate feasibility signal.

**Wave 4 — after Waves 1–3 have soaked in real runs:**
11. R13 (repro/fix-edit overlap), R14 (report pre-drafting), R15 (confirmation rounds) — the three that trade orchestrator simplicity for latency; validate against fresh transcript mining first.
12. R16 (resume) — largest effort, interruption-only benefit; do last.
13. R17 (stall detection) — only after R8 lands and the non-TTY streaming behavior is empirically confirmed.

Dependencies in one line: R17→R8; R12→R4; R15 inlines its own risky-surface guard (no hard dependency on R9); everything else is independent.

## 4. Considered and rejected

No whole proposal was killed — all 16 survived verification — but the skeptic passes cut or replaced significant components. For the record:

- **M1's post-start comment overlap** — saves 2–7s (<1% of a run) while adding a kill-the-reviewer-wave abort branch interacting with post-start's two existing failure modes; bad complexity/latency trade. Post-start stays serial.
- **M1's "collect baseline at §4" join point** — §3's deletion red/green judgment already consumes the baseline; joining at §4 allows two overlapping suite executions on slow suites. Joined before §3 instead.
- **M2's "fixes an existing latent race" justification** — no such race exists; the current barrier already sequences verdicts before edits. The worktree is needed only *because* the amendment introduces concurrency.
- **M3's same-worktree gate/reviewer overlap** — reviewer-run tests colliding with the gate suite can flip it spuriously red and revert good fix commits; replaced with the isolated-checkout precondition.
- **M4's min(reviewer_timeout, 600s) adversary clamp** — contradicts the >5000-line doubling rule; would starve adversaries on exactly the runs that mandate full mode.
- **M4/M8's blanket wakeup/timer ban as originally worded** — would make the Agent-tool deadline unimplementable, reproducing the exact hang being fixed; one one-shot alarm per batch is permitted instead.
- **M6's commit-age freshness heuristic** — tip-commit age says nothing about staleness relative to the remote; replaced with an unconditional fetch.
- **M6's auto-rebase on mid-run author push** — conflict auto-resolution risks silently mangling the author's code or verified fixes; replaced with stop-and-surface, commits preserved.
- **M7's retry of the "unparseable findings" edge case** — that's a successful full-length run with bad formatting; retrying it would double the wave.
- **M8's `.timedout`-stamp discriminator** — the stamp never persists and is never created in the host-kill scenario; replaced with the missing-STATUS-line signature.
- **M9's in-repo state trust and resume-of-judgments** — repo-resident state re-read across sessions is a gate-bypass channel, and resuming recorded verdicts trusts inputs (dismissed list, PR head, adversarial state) that legally move mid-window. Only tamper-evident artifacts are reused.
- **M10's headline 3–7 min saving** — PRIOR ROUNDS already suppresses addressed/deferred re-raises, and the mining evidence cited actually says cross-round dedup *works*; only the refuted-verdict leak is real. Honest saving ~0–3.5 min.
- **M11's rounds=2/max_rounds tiering** — convergence early-exit already gives converging small diffs a 1–2 round fast path; the cut only bites non-converging runs, exactly where a third pass catches second-order bugs, and the adaptive "safety valve" is mechanically disabled by the very reviewer failures that make signal weakest.
- **M12's bare `rounds + budget` formula** — would *grant* new adaptive rounds to `rounds ≥ 6` runs that today get zero; wrapped in `min()` with the old cap.
- **M13's 300s value** — only 1.4× the observed legitimate max from a heavy-tailed distribution; 420s adopted, adaptive rounds exempted.
- **M14's dual hunk-scoped reviewers** — would leave no full-scope reviewer in the exact round whose clean signal can terminate the run; one slot stays full-scope.
- **M15's whole-batch cap-and-defer** — a hard cap silently blocks deletion/high-risk fixes that *cannot* become actionable without reproduction; replaced with never-capped Tier 1 and parallel chunking, which gets the same wall-clock for tokens instead of verdicts.
